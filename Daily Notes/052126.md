## Decisions
- `cp_level` → `VoltageInstallation` (was `voltage`, which is a small internal signal ~0.077, not mains voltage).
- `_CLOUD_TO_LAN_MULTIPLIERS` table introduced for fields where cloud and LAN disagree on unit prefix: ContractedPower ×100 (cloud encodes as W/100), LightLED ×100 (0-1 fraction → 0-100 %).
- Promoted `_async_route_local_or_cloud` + `_is_cloud_only_device` from private `__init__.py` helpers to public `local_api.async_route_local_or_cloud` / `is_cloud_only_device`. Router now accepts `cloud_call=None` for keywords with no V2C Cloud setter — raises `HomeAssistantError` with a user-facing message in cloud-only mode.
- Wired every writeable entity setter (5 Number + 6 Switch + 2 Select = 13) through the router. Added `client.async_cloud_set_logo_led(device_id, enabled)` after a direct probe confirmed `/device/logo_led` exists (HTTP 200) even though the V2C API docs don't list it.
- Hook policy: `PreToolUse:Bash` now exempts `.env*.{example,sample,template,dist}` filenames from the `.env` git-add/cat block via a new `references_unsafe_env()` helper. Templates contain no real secrets and are conventionally safe to commit.
- The `/config` directory is now fully gitignored (was partially tracked via a `!config/configuration.yaml` exception); the user's local HA dev config no longer leaks into commits.
- For LightLED + ContractedPower we don't try to discover/guess undocumented cloud endpoints further: both `/device/light_led` and `/device/contracted_power` returned HTTP 404. Documented as cloud-only-mode limitation in CHANGELOG.

## Meetings & Conversations
- User flagged "LightLED not working — V2C app shows 100 %, HA shows 1.0". Diagnosed live (cloud `light_led: "1.000000"` is a 0-1 fraction, not a percentage). Fix: ×100 multiplier in synthesis.
- User corrected "ContractedPower in V2C app is 0.7 kW, HA shows 5 kW after the last fix." Diagnosed — my earlier ×1000 multiplier was off by 10×; cloud encodes contract_power as W/100, so the correct multiplier is ×100 (cloud "7" → 700 W → entity divides by 1000 → 0.7 kW).
- User followed up: "writing in HA doesn't reflect on the V2C app". Root-caused: every Number/Switch/Select setter called `async_write_keyword` directly; LAN write fails in cloud-only mode and there was no cloud fallback. User chose the full smart-router refactor over piecemeal fixes.
- User restarted HA twice from the host (`docker compose restart homeassistant`) to pick up bytecode changes — `restart: "no"` on the HA container means the in-container Python module cache is the bottleneck for code changes.
